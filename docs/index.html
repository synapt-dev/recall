Memory and coordination for AI coding teams

Synapt turns disconnected AI sessions into an operational system: persistent recall, shared agent channels, durable decisions, and one local-first memory layer your team can actually inspect.

Get started GitHub
$ pip install synapt && claude mcp add synapt -- synapt server && synapt init

Features

Search

Hybrid RRF search

BM25 full-text search fused with 384-dim semantic embeddings via Reciprocal Rank Fusion. Surfaces results that keyword search alone would miss.

Search

Query intent routing

Classifies queries as factual, temporal, debug, decision, aggregation, or procedural. Adjusts recency decay, knowledge boost, and embedding weight automatically.

Memory

Knowledge consolidation

Extracts durable facts from session journals into a knowledge base with lineage tracking, contradiction detection, and confidence scoring.

Memory

Session journal

Rich entries with focus, decisions, accomplishments, and next steps. Auto-extracted at session end, searchable across all sessions.

Memory

Topic clustering

Groups related chunks across sessions via Jaccard token-overlap. LLM-generated summaries for fast context retrieval in concise mode.

Tools

18 MCP tools

Search, journal, reminders, channels, archive backup/restore, and knowledge management. Works with Claude Code, Codex CLI, OpenCode, and other MCP-compatible assistants.

Local-first

Zero cloud dependencies

SQLite + local embeddings. No API keys, no external services. All data stays on your machine. Works offline.

Search

Adaptive working memory

Access tracking boosts frequently-retrieved chunks. Active topics surface naturally without manual curation.

Tools

Operational memory

Shared channels, directives, reminders, and task claims let multiple agents coordinate through one persistent record instead of ad hoc chat logs.

How search works

Three retrieval paths run in parallel and merge at the rank level — no score normalization needed.

query | +--------------+ | intent classify | +--------------+ / | \ half_life emb_wt know_boost / | \ +---------+ +----------+ +-------------+ | BM25/FTS5 | | Embedding | | Knowledge | | + decay | | 384-dim | | FTS + embed | +---------+ +----------+ +-------------+ \ / | +------------------+ | | Reciprocal Rank | | | Fusion (k=60) | | +------------------+ | | | +------------------+ | | cross-encoder | | | rerank (top 20) | | +------------------+ | | / +------------------+ / | cross-session | / | link expansion | / +------------------+ / \ / +---------------------------+ | interleave + dedup + boost | | + adaptive per-session cap | +---------------------------+ | +-----------------+ | ranked context | +-----------------+

Pipeline features

  • Sub-chunk splitting at tool-use boundaries
  • Cross-session link expansion via pre-computed cosine neighbors
  • Adaptive proportional per-session cap
  • Content-aware adaptive filtering (code/personal/mixed)

Token efficiency

Most memory systems inject all memories into every prompt. Synapt lets the LLM decide when to search — so you only pay for context you actually need.

Context stuffing (Mem0, etc.)

Turn 1: system[memories] + question
Turn 2: system[memories] + history + question
Turn 3: system[memories] + history + question
...
Turn N: system[memories] + history + question

Memories repeat every turn. Cost grows linearly with session length. Memory updates invalidate prompt cache.

On-demand search (synapt)

Turn 1: system + question
Turn 2: system + history + question
Turn 3: system + history + question + recall_search
...
Turn N: system + history + question

The LLM chooses when to search. Most turns have zero memory overhead. Tiny stable system prompt always caches.

Benchmarked on LOCOMO conversations with Claude Haiku. synapt v0.7.4, mem0 v1.0.5.
