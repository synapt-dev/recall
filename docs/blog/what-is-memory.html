What Is Memory?

We built an agent memory system from scratch. Here's what we learned.

Layne Penney & Claude · March 2026

I kept losing context. Every time I started a new Claude Code session, my assistant had no idea what we'd been working on. I'd re-explain the project structure, re-state conventions, re-discover decisions we'd already made together. It felt like onboarding the same person every morning.

The worst part was the subtle stuff. We'd debug a tricky race condition, figure out the root cause, settle on a fix — and two days later I'd hit something related and my assistant would suggest the exact approach we'd already ruled out. Not because it was wrong, but because it genuinely didn't know.

So we built a memory system. Not because we planned to — because the frustration got bad enough that we had to. This post is about what we learned along the way. What "memory" actually means, why the obvious approaches don't work, and what does.

Memory is not chat history

The first thing we tried was the obvious thing: save the conversation logs and search them later. BM25 over raw transcripts, a few thousand chunks in a SQLite FTS5 index, done.

It worked — barely. You could find things if you remembered the exact words. But a few weeks of coding sessions is a lot of text. Most of it is noise: file reads, exploratory dead ends, tool output, backtracking. The actual decisions and discoveries are maybe 5% of the total volume, buried in a sea of cat output and abandoned approaches.

The real problem isn't storage. It's retrieval. Having a hundred thousand tokens of past context is useless if you can't find the ten tokens that matter for the question being asked right now.

The two retrieval failures

We learned this the hard way by watching our own searches fail in different directions.

Keyword search misses meaning. Searching for "database connection error" won't find a chunk that says "postgres refused the TCP handshake on port 5432." The information is there, the words aren't. FTS5 is great for exact terms — config keys, error codes, file paths — but useless for paraphrased queries.

Embedding search misses specifics. We added semantic embeddings (384-dim vectors from all-MiniLM-L6-v2, a 90MB model that runs locally in milliseconds). Now we could find semantically similar chunks. But searching for POSTGRES_MAX_CONNECTIONS would return chunks about database configuration in general — semantically close, but not the one with the actual variable name. Embeddings capture meaning, not tokens.

The answer was to run both and merge. Reciprocal Rank Fusion combines two ranked lists by position rather than score — which matters because BM25 scores and cosine similarities are on completely different scales. A result that's #2 in keyword search and #8 in semantic search is probably relevant, even if neither system alone would surface it. This hybrid approach catches what either one alone would miss.

This is the thing that surprised us most early on: neither retrieval path alone is good enough. Embeddings seem like the modern answer, but they fail on the exact-match queries that developers ask constantly — "what port does X run on?", "what's the env var for Y?", specific error messages. You need both.

Raw chunks aren't enough either

Even with good retrieval, raw transcript chunks are noisy. A single chunk might contain three file reads, a failed approach, and one important decision buried in line 47 of the assistant's response. If you return that whole chunk as context, most of it is irrelevant — and in a tight context window, noise actively hurts.

This is where knowledge nodes came in. The idea: run a small LLM over session journals and extract durable facts — things that are true beyond a single session. Not "we discussed the auth system" but "the auth service uses JWT with RS256, keys rotated monthly via AWS Secrets Manager." Specific, searchable, attributed back to the source sessions.

We learned quickly that the extraction quality is everything. Our first attempts with a 3B model pulled out generic observations: "the user is working on a Python project." Not helpful. The good extractions capture decisions, rationale, and specific details — the kind of things a teammate would actually remember about your project.

We also learned the hard way that more knowledge nodes isn't better. We ran experiments capping the number of knowledge nodes returned per query: k=3 was the sweet spot. At k=5 the irrelevant nodes started displacing useful transcript chunks. At k=10 performance dropped further. Uncapped, it was worse than having no knowledge nodes at all. Precision beats volume.

Memory should work like memory

Partway through building this, we had a conversation that reframed everything. We'd been thinking about memory as a database problem: store things, query things. But human memory doesn't work that way. Human memory is adaptive. It changes based on what you access.

The analogy maps surprisingly well:

  • Sensory register → raw transcript JSONL. High volume, ephemeral, unprocessed.
  • Working memory → recently accessed items. Small capacity, fast retrieval.
  • Short-term consolidation → clusters and summaries. Rehearsal promotes items here.
  • Long-term storage → knowledge nodes. Durable facts with confidence scores.
  • Forgetting curve → recency decay. Items not accessed fade exponentially.

The key insight: retrieval is not read-only. Every search should be a write operation too — recording what was accessed, reinforcing it, potentially upgrading its storage quality. A fact you look up every week should be treated differently from one that was mentioned once six months ago.

This led us to design an adaptive layer where access patterns influence what gets promoted, summarized, and retained. Not all of it is built yet, but the architecture now assumes that the memory system evolves over time rather than being a static index.

Not all queries are the same

Another thing we didn't appreciate initially: different questions need different retrieval strategies.

"Why is the worker crashing?" is a debug query — you want recent sessions, aggressive recency decay, raw transcript chunks with error traces. "What database does service X use?" is a factual lookup — you want knowledge nodes, not raw transcripts, and recency doesn't matter much. "What did we change last week?" is temporal — you need date-range filtering, not semantic similarity.

We added a query intent classifier that routes incoming queries and adjusts search parameters automatically. It's lightweight — mostly pattern matching and keyword heuristics, not another LLM call. But it makes a real difference. Debug queries get short half-lives (favor recent context). Factual queries boost knowledge nodes. Temporal queries filter by date range.

The pipeline

Here's what the full system looks like, from raw transcripts to searchable memory:

Raw transcripts (Claude Code JSONL) | Chunking — split into turn-aligned segments with metadata | Embedding — 384-dim vectors for semantic search (90MB model, local) | FTS indexing — SQLite FTS5 for keyword search | Clustering — group related chunks across sessions by topic | Journal extraction — LLM summarizes each session's focus, decisions, outcomes | Knowledge consolidation — distill durable facts with provenance + confidence | Hybrid retrieval — RRF fusion of FTS + embeddings + knowledge search | Ranked, attributed results

Every stage runs locally. Embeddings come from a 90MB model. Enrichment and consolidation use a 3B model via MLX on Apple Silicon or Ollama on Linux. No API keys, no cloud calls. Everything stays on your machine.

What we got wrong

We're being honest about this because the "memory for AI" space is full of inflated claims and unverifiable benchmarks. Here's what actually went wrong:

  • Knowledge extraction is fragile. Small models extract generic observations ("the user is working on code") instead of specific decisions. They sometimes fabricate details that weren't in the source. Our consolidation pipeline once produced cluster summaries that confidently described things nobody actually said.
  • Deduplication is hard. The same fact shows up in slightly different forms across sessions. "We use pytest" and "the test framework is pytest" are the same knowledge node, but detecting that automatically without an LLM call per pair is nontrivial. Jaccard similarity on tokens gets you partway there. It doesn't get you all the way.
  • Source expansion can backfire. We added source attribution so knowledge nodes link back to raw transcript chunks. Good idea — but when uncapped, N knowledge nodes pull in N×3 source chunks, flooding the results with raw context and displacing the actual ranked matches. We saw a 31 percentage point regression before we figured out why. Capping source expansion to the top-N nodes fixed it.
  • Multi-hop reasoning is the hardest category. "Which services would be affected if we changed the auth provider?" requires connecting facts across multiple knowledge nodes. Our single-hop accuracy is decent. Multi-hop is our weakest category. This isn't unique to us — it's the hardest category for every system we've benchmarked against.

Why local matters

Most memory systems in this space — Mem0, Zep, Letta — route through cloud APIs. Your conversation history goes to an external service, gets processed by GPT-4 or similar, and comes back as structured memory. This works, but it means:

  • Your proprietary code, architecture decisions, and internal context leave your machine
  • Every memory operation costs money on top of what you already pay for the assistant
  • You're dependent on API availability, rate limits, and pricing changes
  • It doesn't work offline

We wanted memory to be infrastructure you own, not a service you rent. A SQLite file on disk that travels with your project. Search that returns in microseconds, not hundreds of milliseconds. Enrichment that runs on your hardware without sending code context over the wire.

The tradeoff is real — a 3B model can't match GPT-4 at knowledge extraction. But memory operations are structured tasks: summarize a session, extract facts, classify a query. Small models handle these well enough. And "well enough, instantly, privately, for free" beats "slightly better, with latency, leaking context, for a fee" in most real-world scenarios.

The name "synapt" is a coined truncation of "synapse" — the connection between neurons where signals are transmitted. It evokes "apt" (capable) and "snap" (fast). We wanted a name that felt like what the system does: form connections between scattered pieces of context, quickly.

Where this is going

Memory for AI coding assistants is a new problem space. The benchmarks are immature (everyone uses different metrics and methodologies). The architectures are converging on similar patterns but nobody has solved it. We're still learning.

The things we're most excited about: temporal anchoring (resolving "yesterday" and "last week" to absolute dates at ingestion time), access-driven promotion (frequently retrieved knowledge gets reinforced), and purpose-built extraction models (a small model fine-tuned specifically for knowledge extraction instead of a general-purpose 3B).

But the core philosophy won't change: memory should run on your machine, improve over time, and get out of your way. You shouldn't have to think about it. You should just notice that your assistant remembers things it didn't used to.
